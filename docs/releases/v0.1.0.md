# JiSpec v0.1.0 发布说明

发布日期：2026-04-29

## 发布定位

JiSpec v0.1.0 是一个范围明确的 V1 主线版本，面向“人类审阅在环、契约驱动”的 AI 原生软件交付。

这个版本证明的产品主线是：

```text
bootstrap init-project -> bootstrap discover -> bootstrap draft -> adopt -> verify -> ci:verify -> change -> implement -> verify
```

正确的发布口径是：

- V1 主线版本
- 人类引导的旧仓库接管
- 本地优先的确定性验证与 CI gate
- Greenfield 初始化基础能力
- release / baseline 图谱基础能力

这个版本不是“全自动契约生成”或“自治代码实现”版本。

## 本版包含

### V1 主线

- 稳定的 bootstrap、adopt、verify、CI verify、change、implement 和 `doctor mainline` CLI 命令面。
- 覆盖 takeover、verify、CI verify、fast-lane change、implement handoff 和 post-implement verify 的黄金路径 E2E。
- V1 稳定契约文档：`docs/v1-mainline-stable-contract.md`。
- 面向主线 readiness 的 `jispec-cli doctor mainline`。

### Bootstrap Takeover 强化

- 面向 cache、vendor、audit mirror、build 和 generated-noise 资产的路径级排除策略。
- 拆分 bootstrap inventory 产物：
  - `.spec/facts/bootstrap/evidence-graph.json`
  - `.spec/facts/bootstrap/full-inventory.json`
  - `.spec/facts/bootstrap/adoption-ranked-evidence.json`
- 面向业务、API、schema、document、proto 和 service surface 的 ranked evidence scoring。
- domain re-anchoring、多语言业务词汇提取、aggregate root synthesis、proto domain mapping 和技术边界抑制。
- 人类可读 takeover brief。
- 基于真实旧仓库发现沉淀的 retakeover 回归夹具。

### Greenfield 初始化基础

- PRD / technical solution 输入契约。
- Greenfield 项目资产生成：project framing、domain context draft、API contract、behavior scenario、slice queue、policy、CI gate、review pack 和 AI implementation handoff。
- review workflow：adopt、reject、defer、waive 和中英文 brief 生成。
- Spec Delta、blast radius、provenance drift、two-way ratchet、spec debt、deterministic contract graph 和 static collector 基础能力。

### Release 与 Baseline 基础

- baseline snapshot 与 release compare。
- Merkle Contract DAG，用于确定性 contract graph 快照和版本比较。
- release snapshot 集成 static collector manifest。

## 已知边界

- `implement` 当前证明的是 orchestration、handoff、lane semantics、preflight behavior 和 post-implement verify，不是生产可用的自治补丁生成器。Post-v1 P2-T2 已将 mediation outcome 收敛为 `preflight_passed`、`external_patch_received`、`patch_verified`、`patch_rejected_out_of_scope`、`budget_exhausted`、`stall_detected`、`verify_blocked`。
- 复杂旧仓库在 adopt 阶段仍需要人类审阅和修正。
- LLM provider 可以辅助 draft 和 semantic re-anchoring，但 blocking gate 必须保持确定性。
- Console UI、distributed execution、collaboration / presence surface，以及 direct LLM blocking，都不属于 V1 主线发布契约。

## 后续建议

- 强化 `implement` 背后的实际 patch loop。Post-v1 P2-T1 已落为 external patch mediation。
- 优化 preflight pass 场景下的 implement outcome 命名。Post-v1 P2-T2 已关闭该命名缺口。
- 持续补强真实仓库 takeover 回归夹具。
- 继续把 `doctor mainline`、`regression-runner` 和 `ci:verify` 作为后续变更的 release gate。
